🛡 Safety Middleware — 5 Gates, 3 Tiers, 2 Layers
Every AI response passes through 5 independent safety gates before reaching a clinician or patient. Any single gate can block. The system defaults to the most restrictive safe state on any error — conservative by design.

💡 Safety Pipeline Flow

Patient Input → Gate 1: EligibilityEngine → Gate 2: SafetyValidator (3-Tier) → Gate 3: ContentSafetyValidator (2-Layer) → Gate 4: OutputFilteringService → Gate 5: 7-Agent Byzantine Consensus → Safe Output to Clinician
1

EligibilityEngine — Pre-Consultation Hard Stops

File: backend/src/services/weight-loss/EligibilityEngine.js
Hard stops for: MTC/MEN2 history, SCOFF eating disorder screen (≥2 = blocked), PHQ-9 Q9 suicidal ideation (triggers crisis protocol with Lifeline, 000, Beyond Blue, Kids Helpline, MensLine), BMI thresholds with ethnicity adjustment (Asian/South Asian/Middle Eastern/Aboriginal & Torres Strait Islander), age 18–75, Australian state S4 regulations. Runs before any AI interaction.

2

SafetyValidator — 3-Tier Hybrid Drug Safety (9 Parallel Checks)

File: backend/src/middleware/SafetyValidator.js
Tier 1 (Deterministic, <50ms, FREE): Queries 3,778 Supabase safety records across 8 tables — drug interactions (501), contraindications (773), TGA pregnancy categories (1,699), dosing guidelines (755), allergy cross-reactions (50), food interactions (225), supplement interactions (150), QT prolongation risk. Uses node-cache (15-min TTL) for sub-millisecond repeat lookups.
Tier 2 (AI, ~2s, $0.0016): Claude Haiku 4.5 handles novel drugs, complex multi-drug scenarios, and edge cases not in the deterministic database. Structured JSON output with confidence scores (85–92%).
Tier 3 (Conservative, <10ms, FREE): If Tiers 1 and 2 both fail, returns: "Insufficient safety data — consult specialist before prescribing." Never passes through an unchecked prescription.

3

ContentSafetyValidator — Dual-Layer Real-Time AI Response Validation

File: backend/src/middleware/ContentSafetyValidator.js
Layer 1 (Real-time streaming, <100ms overhead): Scans every 50 tokens during Claude streaming for: red flag symptoms without emergency referral, contraindicated medication mentions, missed urgent symptoms. Runs in parallel with AI generation — doesn't slow down response time.
Layer 2 (Post-stream deep validation, 2–5s, async): After AI response completes, Haiku 4.5 performs 5-dimension deep validation: medication safety, diagnosis completeness, urgency appropriateness, differential completeness, referral appropriateness. If critical issues found, Haiku generates a corrected replacement response.
Both layers use 3-tier architecture (deterministic → Haiku AI → conservative fallback).

4

OutputFilteringService — AHPRA/TGA Language Compliance

File: backend/src/services/weight-loss/OutputFilteringService.js
Regex-based blocking of 4 categories of forbidden language:
• Diagnostic claims: "I diagnose you with", "your diagnosis is", "my diagnosis is"
• Definitive claims: "you have diabetes", "you definitely have", "there is no doubt"
• Prescription claims: "I am prescribing", "here is your prescription", "take this medication"
• Treatment decisions: "you must start treatment", "begin taking this medication"
Each blocked phrase is replaced with clinician-deferral alternatives (e.g. "A doctor will assess" or "Your healthcare provider will determine").

5

7-Agent Byzantine Consensus — Multi-Provider Prescription Validation

File: backend/src/services/42-agent-prompts-v2.js, backend/src/services/safetyTeamValidator.js
7 specialised AI agents from 4 different providers must reach consensus before any prescription is approved. Temperature locked at 0.2 — "safety cannot be creative."
Agents: Drug Interaction Validator, Allergy Checker, Contraindication Enforcer, Dose Validator, Pregnancy/Breastfeeding Safety Officer, TGA Compliance Auditor, Safety Consensus Lead.
Providers: Claude Opus 4.6 (Anthropic), Gemini 3 Flash (Google), GPT-5.2 (OpenAI), Perplexity Sonar Pro.
ANY 1 agent can VETO. Single-model bias is eliminated through provider diversity.

👤 Data Anonymisation — 5-Layer PHI Protection Stack
Patient data is protected by 5 independent anonymisation layers, each using different technology. Even if one layer fails, the others catch it. The stack includes Microsoft Presidio (open-source NLP-based PII detection), custom regex sanitisers, and OTEL-level scrubbing.

💡 Anonymisation Pipeline

Raw Patient Data → Layer 1: Claude AI Anonymizer → Layer 2: Microsoft Presidio → Layer 3: Audit Logger PII Masking → Layer 4: OTEL Span Sanitizer → Layer 5: Encryption Log Sanitizer → PHI-Free Output

1️⃣ Layer 1: Claude AI Anonymizer

File: backend/src/utils/claude-ai-anonymizer.js

StripsName, email, phone, specific address
PreservesALL medical context: symptoms, history, meds, allergies, vitals, age, gender
MethodStructural de-identification — builds anonymised context object, never passes raw patient record to Claude
Location generalisationSpecific city → region (e.g. "Adelaide" → "Australia")
Patient IDTruncated UUID (first 8 chars only)

Principle: AI doesn't need identity for clinical reasoning. 100% medical quality preserved, 0% PII transmitted.

2️⃣ Layer 2: Microsoft Presidio

Standard HIPAA entities: PERSON, EMAIL_ADDRESS, PHONE_NUMBER, LOCATION, DATE_TIME, IP_ADDRESS, URL, MEDICAL_LICENSE, US_SSN

Custom Australian recognisers (5):

AU_MEDICAREMedicare card numbers (10–11 digits, starts with 2–6)

Architecture: Node.js client spawns Python subprocess. 3-second timeout, 1 retry, result caching (1hr TTL, max 1,000 entries). 95% latency reduction via cache. Graceful fallback: if Presidio is unavailable, text is redacted entirely (safe default).

⛓ Audit Trail & SHA-256 Chain Hashing
Every clinical decision is recorded in an append-only, SHA-256 chain-hashed audit trail (blockchain-style). Each record cryptographically links to the previous via hash — tamper-evident by design. Database triggers enforce immutability at the PostgreSQL level.

⛓ Chain Hashing Architecture

File: backend/src/services/weight-loss/ClinicalDecisionAuditService.js

AlgorithmSHA-256 (Node.js crypto module)
Genesis hashgenesis:botaniqal:tga-cdss-audit:2026
Chain formulaSHA-256(prevHash + consultationId + patientId + eventType + timestamp)
SequenceAuto-incrementing sequence_number for ordering
VerificationverifyChain(fromDate, toDate) walks chain and validates each link
ImmutabilityPostgreSQL triggers prevent UPDATE/DELETE on clinical_decision_audit and adverse_event_reports

  prevHash, consultationId, patientId,
